# Band Guidance

> **⚠️ Experimental feature — off by default.** Band plans are voluminous,
> region-specific, and genuinely hard to get exactly right. This feature is
> shipped as experimental and disabled by default; enable it only if you find it
> useful, and treat every suggestion as a hint, not gospel. Corrections to the
> data are welcome via the issue tracker.

Band Guidance is an optional, **advisory** convenience feature that helps
operators — especially newcomers to HF digital modes — find the right place to
operate. It offers **defaults and warnings, never hard locks**. It can suggest a
mode's "watering hole" and warn when a frequency falls outside the customary
digital segment, but it never prevents the operator from doing anything.

> Amateur band plans are voluntary, are **not** regulations, and differ by IARU
> region. Band Guidance reflects customary usage to reduce confusion; the
> operator is always responsible for lawful operation in their own country.

## Design philosophy

- **Guide, don't gate.** Every suggestion is informational. If the operator
  ignores it, nothing changes.
- **Quiet by default, loud only when useful.** Guidance fires on the two moments
  that matter (mode/band change, frequency change), not on every action.
- **Region-aware, not location-aware.** We never infer the operator's location.
  The region is a user setting; we cannot and do not verify it.
- **Data, not code.** All frequencies live in a curated, documented data file so
  they can be reviewed and updated without touching logic.

## Configuration

Two new settings, both surfaced in the connector settings form and as
environment variables:

| Setting | Env var | Default | Notes |
| --- | --- | --- | --- |
| Band Guidance | `FLDIGI_BAND_GUIDANCE` | `off` | Master switch for the whole feature. Off = silence. **Experimental.** |
| IARU Region | `FLDIGI_REGION` | `2` | `1` = Europe/Africa/N.Asia, `2` = Americas, `3` = Asia-Pacific. |

Default is **off**: the feature is experimental and the band-plan data is hard
to get perfectly right, so it stays silent until the operator deliberately turns
it on. Once on, it can be switched back off in one click.

## Behavior

Band Guidance hooks the two control tools and adds advisory information to their
results. The assistant relays the guidance and asks the operator what to do; the
tool itself never blocks.

### A. Mode change or band change → watering-hole suggestion

Triggered when the modem changes (`set_modem`) **or** when a frequency change
crosses into a different band. Logic:

1. Determine the current band from the frequency.
2. Look up the preferred ("watering hole") frequency for the new mode on that
   band, for the configured region.
3. If the current frequency is **not at** that watering hole (outside a small
   tolerance), return a suggestion: *"The PSK31 watering hole on 20 m is
   14.070 MHz; you're at 14.085. Move there?"*
4. The assistant asks. If the operator declines or ignores it, the frequency is
   left untouched.

This fires **once** per mode/band change — not repeatedly.

### B. Frequency change → band-plan segment check

Triggered on `set_frequency`. Logic:

1. Determine the band and the customary segment for the **mode's category**
   (see Mode categories) for the configured region.
2. If the new frequency falls **outside** that segment, return a warning quoting
   what the band plan says for that band: *"14.350 is outside the 20 m
   RTTY/Data segment (14.070–14.095 in Region 2). Proceeding anyway is your
   call."*
3. No change is reverted; this is information only.

### CW is a digital mode too

CW is generated by digital software but operates in the **CW allocations**, not
the RTTY/Data segment. The band-plan check keys off the mode's *category*: a CW
mode is checked against the CW segment, a data mode against the RTTY/Data
segment. See Mode categories below.

## Mode categories

fldigi exposes many modems. For guidance we map each to a category that selects
the right band-plan segment. The mapping is data-driven with a safe default.

| Category | Example fldigi modems | Checked against |
| --- | --- | --- |
| `CW` | CW | CW segment |
| `DATA` | BPSK31/63/125, RTTY, Olivia, Contestia, MT63, MFSK, THOR, DominoEX, Olivia | RTTY/Data segment |
| `DATA_FT` | FT8, FT4, JT65, JT9 (where present) | Data segment (FT sub-band) |
| `IMAGE` | MFSK image, SSTV, WEFAX | SSTV/image conventions |
| _default_ | anything unmapped | `DATA` (most conservative for digital) |

## Tools (MCP surface)

Band Guidance is exposed both passively (enriched results) and actively
(query tools), so the assistant can reason about it without ever forcing a change.

- **`set_modem`** / **`set_frequency`** — results gain an optional `guidance`
  object: `{ "kind": "watering_hole" | "band_plan", "message": "...",
  "suggested_hz": 14070000, "segment": "14.070–14.095" }`. Absent when nothing
  to say or Band Guidance is off.
- **`band_plan_lookup(frequency_hz, mode?)`** — what the band plan says at a
  frequency for the region: band name, segment, category, whether in-segment.
- **`watering_hole(mode, band?)`** — the preferred frequency for a mode on a
  band (defaults to the current band).
- **`move_to_watering_hole(mode)`** — convenience that sets the mode and the
  preferred frequency together (still subject to the transmit gate before any
  transmit; this only changes settings, it does not key the radio).

The "ask, and leave alone if declined" loop lives in the conversation, which is
exactly where soft guidance belongs.

## Data model

A single curated file, e.g. `src/fldigi_mcp/data/band_plans.yaml`, versioned and
documented:

```yaml
regions:
  "2":
    name: "IARU Region 2 (Americas)"
    source: "ARRL Considerate Operator's Frequency Guide (QST, Nov 2013)"
    bands:
      20m:
        edges_hz: [14000000, 14350000]
        segments:
          - { category: CW,   range_hz: [14000000, 14150000] }
          - { category: DATA, range_hz: [14070000, 14095000], note: "RTTY/Data" }
          - { category: DATA, range_hz: [14095000, 14099500], note: "auto data" }
    watering_holes:
      20m: { PSK31: 14070000, RTTY: 14080000, FT8: 14074000, ... }
```

Each region carries an explicit `source` and date so the data is auditable. The
loader validates ranges on startup.

## Data coverage

Curated and validated in `band_plans.yaml`, each region transcribed from a
**named, dated source** (no guessed numbers):

- **CW and digital segments** for **160 m – 10 m** in all three regions, plus
  **6 m / 2 m / 70 cm** in all three regions.
- **93 watering-hole frequencies** across 10 modes (FT8, FT4, JT65, JT9, WSPR,
  PSK31, RTTY, Olivia, MT63, JS8Call).
- A validation script checks that every segment is ordered, sits within its band
  edges, and that every watering hole lands in-band.

Known caveats (documented rather than hidden):

- Bands above 70 cm (microwave) are out of scope — not where fldigi digital
  text modes operate.
- A few VHF/UHF calling frequencies are region-specific (e.g. 70 cm FT8 is
  432.174 in R1 vs ~432.065 in R2); the globally-harmonised ones (6 m 50.313,
  2 m 144.174) are in the shared table, the rest are left to the segment check.
- Band plans are revised periodically; the data carries a date and source per
  region so it can be refreshed.

Sources used (all from the IARU band-plan hub
<https://www.iaru.org/reference/band-plans-2/>):

- **Region 1**: IARU R1 HF Band Plan (eff. 16 Oct 2020) + IARU R1
  VHF/UHF/Microwave Band Plan.
- **Region 2**: ARRL Considerate Operator's Frequency Guide + ARRL band plan
  (VHF/UHF), US usage; harmonised with the IARU R2 plan.
- **Region 3**: IARU R3 Interim Band Plan (doc R3-004).

Until a region/band is curated, Band Guidance degrades gracefully: it offers
watering-hole suggestions where it has data and stays silent (rather than
guessing) where it does not.
